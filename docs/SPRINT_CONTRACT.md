# SPRINT CONTRACT

各 TASK を始める前に、Generator はこのルールを確認してから実装を開始します。

## 契約の目的

- 「いい感じに実装した」で完了にしない
- Evaluator が客観的に確認できる完了条件を持つ
- 1 TASK の範囲を明確にして、過剰実装を防ぐ

## Generator が TASK を始める前に確認すること

### 1. TASK の完了条件を読む

`docs/MVP_TASKS.md` の対象 TASK の「完了条件」を必ず読む。

完了条件が曖昧な場合（「いい感じに」「適当に」「後で」など）は実装を始めず、Planner に差し戻す。

### 2. REQUIREMENTS.md との整合を確認する

`docs/REQUIREMENTS.md` の該当機能・画面の要件を確認する。
矛盾がある場合は人間に確認する。

### 3. STATUS.md の依存 TASK が完了しているか確認する

依存 TASK が DONE でない場合、その TASK を先に完了させる（または人間に相談する）。

## 完了条件の書き方ルール（Planner 向け）

完了条件は **Evaluator がブラウザ操作で確認できる形** で書く。

### OK の例
- 「ホーム画面を開くと〇〇が表示される」
- 「〇〇ボタンをクリックすると〇〇画面に遷移する」
- 「〇〇の入力欄に文字を入れると〇〇が更新される」
- 「スマホ幅（375px）で表示が崩れない」
- 「`npm run build` が成功する」

### NG の例
- 「いい感じのUIにする」→ 基準が曖昧
- 「後で詰める」→ 完了条件ではない
- 「コードを整理する」→ 何が完了かわからない
- 「動くようにする」→ 何が動けばいいか不明

## TASK 不合格時のルール

Evaluator が不合格と判定した場合:

1. Evaluator は **修正プロンプト** を作成して Generator に戻す
2. Generator は修正プロンプトの指示のみを実装する
3. 修正後、再度 Evaluator が評価する
4. 3回不合格が続く場合は人間に報告する

## 自己評価の禁止

Generator の自己評価だけで TASK を「完了」扱いにしてはならない。
必ず Evaluator の評価を経てから `docs/STATUS.md` を DONE にする。

## 一度に実装できる TASK は1つだけ

複数 TASK を同時に実装しない。
「ついでに」「まとめて」は禁止。
1 TASK = 1 実装 = 1 commit。
